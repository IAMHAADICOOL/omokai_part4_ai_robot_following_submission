# Our changes to tb3_multi_robot

This package is a **vendored fork** of
[github.com/arshadlab/tb3_multi_robot](https://github.com/arshadlab/tb3_multi_robot)
by Arshad Mehmood (License: **Apache 2.0** — see `package.xml`). The upstream
project's own documentation is kept unchanged in **`UPSTREAM_README.md`**.

We include it directly in `src/` (rather than cloning it at build time) so the
build needs no network and always uses this exact, tested version.

## What we changed and why

- **`models/turtlebot3_burger/model.sdf`** — `odom_publisher_frequency` raised
  **5 → 30 Hz**. Nav2's controller/costmap need fresh odometry transforms; at
  5 Hz the transform was stale during motion, which smeared the local costmap.

- **`launch/tb3_nav2.launch.py`** — added a **per-robot AMCL auto-seed**. The
  upstream params ship with `set_initial_pose: false`, so AMCL never localized on
  its own. We publish each robot's known spawn pose (from `config/robots.yaml`) to
  `/<namespace>/initialpose` a few seconds after startup, so every robot localizes
  automatically with no manual "2D Pose Estimate."

- **`launch/tb3_world.launch.py`** — adapted for ROS 2 Jazzy / Gazebo Harmonic
  (e.g. `robot_description` passed via `ParameterValue(...)`), plus in-file notes
  from debugging the per-robot TF setup.

## How it's used here
The formation pipeline (`omokai_bringup/formation.launch.py` +
`formation_coordinator`) drives the `/<namespace>/follow_waypoints` actions that
this package's per-robot Nav2 stacks expose. See the repository root README,
Section 1d ("Part 2"), for the launch order.

- **`params/burger_nav2_params.yaml`** (and `burger_cam` variant) —
  `waypoint_pause_duration` **200 → 0**. Upstream made the robot dead-stop for
  200 ms at every waypoint; with several waypoints per robot this made formation
  moves look sluggish and stop-start. 0 = drive through waypoints smoothly.

## Part 4 additions (camera + ArUco marker for the vision follow demo)

Part 4 (`vision_follow`) needs a camera on the robot and something for one robot
to recognise on another. Both live here, in a **new `burger_cam` model variant**,
so plain `burger` (used by Parts 1 & 2) is untouched. `burger_cam` is selected
only by setting `TURTLEBOT3_MODEL=burger_cam` before launching; the default stays
`burger`.

- **`models/turtlebot3_burger_cam/model.sdf`** — a burger with two additions,
  both attached directly to `base_link` (no extra links/fixed-joints, which were
  found to make the physics solver jitter):
  - a **camera sensor** (`640×480`, 15 Hz) publishing `camera/image_raw` and
    `camera/camera_info`;
  - an **ArUco marker** (`DICT_4X4_50`, id 0) as a textured box on the rear of
    the chassis, so a following robot behind it can see and identify it. The
    marker material sets an explicit white base colour so the PBR renderer does
    not tint the texture black, and the texture is referenced with a portable
    `model://turtlebot3_burger_cam/textures/aruco_marker_0.png` URI (resolved via
    `GZ_SIM_RESOURCE_PATH`, not an absolute filesystem path).
- **`models/turtlebot3_burger_cam/textures/aruco_marker_0.png`** — the marker
  image, generated with OpenCV's `cv2.aruco` (see `vision_follow`).
- **`urdf/turtlebot3_burger_cam.urdf`**, **`params/burger_cam_bridge.yaml`**,
  **`params/burger_cam_nav2_params.yaml`** — the `burger_cam` counterparts of the
  existing burger files.
- **`launch/tb3_world.launch.py`** — spawns a separate `ros_gz_image image_bridge`
  for the camera image when the model is not plain `burger` (the raw image is
  bridged by `image_bridge`, while `camera_info` goes through the normal
  `parameter_bridge`).
- **`setup.py`** — the `models/` install rule was made a recursive
  `models/**/*` file glob so new model folders (like `turtlebot3_burger_cam`,
  including its `textures/` subfolder) install automatically.

The marker is generated by, and consumed by, the standalone `vision_follow`
package (`src/vision_follow/`). See its README and the repository root README
Section for how Part 4 is run.
